# How pubdoc works

This document explains how `pubdoc get` provides appropriate version of
documentation for the given package, and how it manages different versions of
documentation for different packages.

## Requirements

pubdoc utilizes the package management mechanism of `dart pub` command to build
the appropriate version of documentation for the packages. So ensure that the
packages are included in the project's dependency and the dependency tree is
up-to-date by running `dart pub get` first.

Note that the packages don't have to be declared in the `pubspec.yaml` directly.
Since pubdoc looks up the package versions in `pubspec.lock`, it is also aware
of indirect dependencies of your project.

## Detect package version

When pubdoc is asked for documentation given a package name, it first resolves
the package name to a version number that your project actually uses. It does
this by looking at the `pubspec.lock` file in your project. This file is
generated when you run `dart pub get` and it would look something like this:

```yaml
# Generated by pub
# See https://dart.dev/tools/pub/glossary#lockfile
packages:
  firebase_core:
    dependency: "direct main"
    description:
      name: firebase_core
      sha256: "3b19a47f6ea7c2632760777c78174f47f6aec1e05f0cd611380d4593b8af1dbc"
      url: "https://pub.dev"
    source: hosted
    version: "4.5.0"
...
```

You can see that the `version` field has the package version that your project
actually depends on, which is `4.5.0` in this case. pubdoc will provide the
documentation compatible with that version throughout the subsequent steps.

## Resolve documentation version

pubdoc does not always generate documentation for every version of the same
package. For example, you have two projects that use firebase_core v4.5.0 and
v4.5.3 for each. Those version numbers differ only in the patch version, so we
can assume that their public API is the same and we can reuse the same
documentation—presumably generated for v4.5.3—for both projects. This way, we
can save time and storage space while still providing up-to-date documentation
for the versions that are actually used in projects.

pubdoc supports several version resolution strategies, which can be configured
via CLI options or in the `.pubdocrc` file:

- `exact`: Provides exact versions of documentation for every package version.
  For example, if one of your projects uses firebase_core v4.5.0 and another one
  uses v4.5.1, then both projects will use different documentation for their
  respective versions. This mode provides always accurate documentation, but
  requires more time and storage space.

- `loose-patch`: Shares the same documentation for projects that use the same
  major and minor version of a package. For example, if one project uses
  firebase_core v4.5.0 and another uses v4.5.4, then both projects will use the
  same v4.5.x documentation. The actual documentation version will be the latest
  one among the projects, which is v4.5.4 in this case. When a project adopts
  firebase_core of a higher patch version (say, v4.5.6), pubdoc will then
  automatically update the shared documentation as well to v4.5.6.

- `loose-minor`: The same rule as the `loose-patch`, but be strict only on the
  major version. For example, if one project uses firebase_core v4.5.0 and
  another uses v4.7.2, both projects will use the same v4.x documentation.

## Manage documentation versions

Once the required documentation version is resolved, pubdoc checks if the
documentation for that version already exists in the cache, and if not, it
generates the documentation and stores it in the shared cache for future use.

### Shared cache

pubdoc stores all generated documentation in a cache directory, which is
typically located at `~/.pubdoc/cache/`. It organizes the documentation by
package name and version number; different versions of documentation for the
same package are placed in the same directory named after the package.

```
~/.pubdoc/cache/
|-- firebase_core/
|   |-- firebase_core-4.5.0/ # documentation for firebase_core v4.5.0
|   `-- firebase_core-3.15.2/
`-- dio/
    |-- dio-5.0.0/
    `-- dio-4.0.6/
```

Each documentation also has a `metadata.json` file in its root directory, which
contains:

- `version`: The documentation version.
- `package_version`: The canonical version number of the package that the
  documentation is generated for. For example, if the documentation is generated
  from firebase_core v4.5.4, then the version field would be `4.5.4`.
- `source`: The actual source location of the package that the documentation is
  generated from.
- `tool_version`: The version of pubdoc that generated the documentation.

```json
{
  "version": "4.5.x",
  "package_version": "4.5.4",
  "source": "file:///Users/username/.pub-cache/hosted/pub.dev/firebase_core-4.5.4",
  "tool_version": "1.0.0"
}
```

### Version management

When and how documentation is created or updated depends on the documentation
version resolution strategy. The simplest case is the `exact` strategy; pubdoc
checks if the documentation for the given package version already exists in the
cache, and if not, it generates the documentation and stores it in the cache.
Otherwise, it reuses the existing one.

For the other strategies, `loose-patch` and `loose-minor`, a single
documentation would cover multiple versions of the same package. In this case,
the documentation is named with a wildcard version number like dio-5.1.x or
dio-4.x, and stored in the same directory as other documentations for canonical
version numbers like dio-5.0.1.

```
~/.pubdoc/cache/
`-- dio/
    |-- dio-5.0.1/ # documentation for dio v5.0.1 only (exact)
    |-- dio-5.2.x/ # documentation for dio v5.2.0, v5.2.1, ... (loose-patch)
    |-- dio-4.x/   # documentation for dio v4.0.6, v4.2.0, ... (loose-minor)
```

For example, say a `loose-*` strategy has resolved the documentation version to
v5.3.x for dio. If it doesn't exist in the cache, pubdoc will generate the
documentation for the required version (say, v5.3.2) and store it in
`~/.pubdoc/cache/dio/dio-5.3.x/`. The `metadata.json` then would look like:

```json
{
  "version": "5.3.x",
  "package_version": "5.3.2",
  "source": "file:///Users/username/.pub-cache/hosted/pub.dev/dio-5.3.2"
}
```

Unlike the case of `exact`, the `loose-*` strategies may also update the
existing documentation if the package version it bases on is lower than the
required version. For example, say we've asked for documentation for dio v5.3.6
with the `loose-patch` strategy, the documentation version is resolved to
v5.3.x, which already exists in the cache:

```
~/.pubdoc/cache/
`-- dio/
    |-- dio-5.3.x/
        |-- metadata.json
```

If the `package_version` in the `metadata.json` is greater than or equal to
v5.3.6, pubdoc simply reuses that documentation without any modification.
Otherwise, pubdoc will regenerate it from the v5.3.6 source of dio and replace
the existing dio-5.3.x documentation with the new one under the same directory
name. Then, the updated `dio-5.3.x/` directory should also has an updated
`metadata.json` like this:

```json
{
  "version": "5.3.x",
  "package_version": "5.3.6",
  "source": "file:///Users/username/.pub-cache/hosted/pub.dev/dio-5.3.6"
}
```

### Project-level documentation library

pubdoc also creates a project-level documentation library under the project's
`.pubdoc/` directory, which contains symlinks to the actual documentation in the
cache. This way, your agents can easily access the appropriate version of
documentation for the project they're working on, without knowing about the
actual version and the cache location.

```
<project-root>/
`-- .pubdoc/
    |-- firebase_core -> ~/.pubdoc/cache/firebase_core/firebase_core-4.5.4/
    |-- dio -> ~/.pubdoc/cache/dio/dio-5.2.x/
    `-- ...
```

This library is automatically managed in a trivial way when running
`pubdoc get`—adds symlinks to the documentation for the requested packages if
they don't exist in `.pubdoc/` yet.

## Generate documentation

pubdoc uses [dartdoc_builder][1] to generate API documentation for Dart/Flutter
packages. Given the path to a Dart package, it analyzes the package's source
code and produces a set of structured text-based documentation files for the
public API. That is to say, pubdoc generates documentation locally on your
machine. But then where does the source code come from?

[1]: https://github.com/fujidaiti/pubdoc/tree/main/dartdoc_builder

Thankfully, `dart pub get` command downloads all the dependencies of your
project and caches them on the disk. It also creates an index of those cached
packages in `.dart_tool/package_config.json`, which would look something like
this:

```json
{
  "configVersion": 2,
  "packages": [
    {
      "name": "firebase_core",
      "rootUri": "file:///Users/username/.pub-cache/hosted/pub.dev/firebase_core-4.5.0",
      "packageUri": "lib/",
      "languageVersion": "3.2"
    },
    ...
  ],
}
```

You can see that the `rootUri` field indicates the location of the cached
package; in this case, the source code of firebase_core v4.5.0 is located in
`~/.pub-cache/hosted/pub.dev/firebase_core-4.5.0`. And this source has the
canonical Dart package structure, so we can run dartdoc_builder on this
directory to generate the documentation.

## Q&As

### Is it safe to remove the cache?

Yes, pubdoc manages cached documentation in a simple way: create if not exist.
So you can freely remove the shared cache directory and `.pubdoc/` per project.

### Can it handle packages that are not hosted on pub.dev?

Yes. pubdoc supports all [Dart dependency sources][2] except for SDK packages,
such as GitHub and local storage.

[2]: https://dart.dev/tools/pub/dependencies#dependency-sources

This is because the `dart pub` command manages project's dependencies from all
those sources in a unified way: package version numbers for the project in
`pubspec.lock`, and its source locations in `package_config.json`. So as long as
the package is properly included in the project, pubdoc can generate
documentation for it regardless of where it is hosted.

### Does it support pub workspaces?

Yes. You can run `pubdoc get` in both the workspace root and individual projects
in the workspace.

Note that, in a workspace, the `.pubdoc/` directory is created only in the
workspace root, and all individual projects share the same `.pubdoc/`. This is
because the workspace projects share the same dependencies, as stated [here][3].

[3]: https://dart.dev/tools/pub/workspaces

### How are pre-release versions handled?

Pre-release versions are always resolved as if the `exact` strategy were in
effect, regardless of the configured resolution strategy. That is, `1.0.0-dev.1`
always resolves to documentation version `1.0.0-dev.1`, never to a shared
version like `1.0.x` or `1.x`, even if `loose-*` strategies are specified.

Quoting the [SemVer specification][4] as to why this is the case:

> A pre-release version indicates that the version is unstable and might not
> satisfy the intended compatibility requirements as denoted by its associated
> normal version.

Note that this rule applies to documentation version resolution only. The rest
of the flow—cache lookup, documentation generation, symlink creation—works the
same way as for stable versions.

The cache directory would look like this:

```
~/.pubdoc/cache/
`-- firebase_core/
    |-- firebase_core-2.0.0-beta.1/  # pre-release, always exact
    |-- firebase_core-2.0.0-beta.2/  # pre-release, always exact
    |-- firebase_core-2.x/           # loose-minor docs, never shared with pre-releases
```

[4]: https://semver.org/#spec-item-9

### When should I run `pubdoc get`?

You don't need to run `pubdoc get` multiple times for the same package unless
`pubspec.lock` changes. It fixes package versions your project uses, so running
the command against the same `pubspec.lock` would always give you the same
documentation for the same package (of course, unless pubdoc configuration
changes).

`pubspec.lock` changes, for example, when you add a new package to your project,
or update version constraints in `pubspec.yaml` and run `dart pub get` again. In
those cases, the package version may change within its constraints, since last
time pubdoc generated documentation for it. So you want to run `pubdoc get`
again to maintain the documentation up-to-date.

Note that, however, pubdoc validates and refreshes the documentation cache
on-demand. So running `pubdoc get` once when you need the documentation would be
just enough.
